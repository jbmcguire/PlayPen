Confirm before playpen://configure changes the publish host

P1SECURITYplan 001

A crafted playpen://configure?service=… deep link silently overwrites the hosted-service URL every future publish/replace/delete is sent to — including the publish token — showing only an after-the-fact alert. A link from a webpage or chat message can reroute a user's artifacts and token to an attacker. Fix: stage the URL and require explicit confirmation showing the destination before persisting.

ContentView.swift:131 — configureHostedService(from:) writes the override with no confirmation
Impact: credential + artifact theft via phishing linkEffort: SRisk: LOWConfidence: HIGH

Restrict playpen://import to http(s) URLs

P2SECURITYplan 002

The import path hands the embedded url straight to the resolver with no scheme check. Today resolve rejects non-HTTP responses so file:/data: fail late — but relying on a downstream type check for a security boundary is fragile. Validate the scheme at the deep-link entry (defense-in-depth).

ContentView.swift:145 — hostedURL(from:) returns URL(string:) unchecked
Impact: latent local-file / data-URI import if resolver changesEffort: SRisk: LOWConfidence: HIGH

Harden public-link derivation against Host-header injection

P2SECURITYplan 003

When PLAYPEN_PUBLIC_BASE_URL is unset, every public link is built from the client-controlled x-forwarded-host / Host header. Behind a permissive proxy an attacker can point those links at their own domain. Preflight only warns. Fix: validate the forwarded host and make --require-public require the env var.

server.js:657 — forwarded host trusted as-is · production-preflight.js:158 — warns, does not fail
Impact: cache-poisoning / wrong links in productionEffort: SRisk: LOWConfidence: MED

Verifier digest diverges from server on whitespace-padded annotations

P2BUG · TECH-DEBTplan 004

The content digest is reimplemented in four places. Three trim the annotation before hashing; verify-host.js does not. A payload whose annotation has leading/trailing whitespace digests differently in the verifier than on the server — so the verifier can reject a correctly-published record. Fix the verifier, write a canonical spec, add a conformance test.

verify-host.js:663 pushes payload.annotation untrimmed vs. server.js:508 / inspect-link.js:311 / HostedPlaygroundService.swift:229 (all trim)
Impact: spurious verify failures; silent ETag driftEffort: MRisk: MEDConfidence: HIGH

Considered & rejected recorded so they aren't re-audited

  • Missing await on S3 deletePayload — the await is present (storage.js:152/164); delete correctly returns false on a missing record. Subagent misread.
  • No SECURITY.md / CODE_OF_CONDUCT.md — both already exist at the repo root.
  • Path traversal in storage / static serving — IDs validated by safeID ([A-Za-z0-9_-]{8,80}); static serving uses path.normalize + startsWith guard + .playpen-store block. Defended.
  • S3 read-error response-body leak — Node GC reclaims the small unconsumed body; not worth the churn.
  • No ESLint/Prettier/typecheck gate — zero-dependency by deliberate design; a linter's dependency cost outweighs the benefit at this size.
  • Token-comparison length-timing leak (server.js:762) — real but very low severity (leaks only token length, only when configured, behind network jitter). Fold into plan 003 if touching that file.
  • Markdown render unbounded by line count — already bounded by the 2 MB publish cap; client-side in the viewer only.
  • S3 driver only tested against a hand-written fake — a real MinIO/S3 test needs Docker in CI, conflicting with the simple-CI posture. Revisit before the first production S3 deploy.
  • No native Swift test target — deliberately deferred per PLAN.md phasing; L-effort new target. Plans 001/002/004 note where tests slot in if one is added.

Direction not planned — maintainer's call

  • MCP server wrapping the existing OpenAPI — the project's whole pitch is "agents publish artifacts here" (AGENTS.md), yet there's no MCP entry point. Strongest-grounded option.
  • npx playpen publish — publish-file.js is already standalone; packaging removes the "clone the repo to publish" barrier for agents in sandboxes.
  • GitHub Action to publish an artifact and comment the link on a PR — fits the stated QA/review audience; ci.yml already exercises the service but nothing consumes it.
