# Implementation Plans

Generated by the improve skill on 2026-06-12, surveying PlayPen for public-release
readiness (the repo just went public at github.com/jbmcguire/PlayPen). Written
against commit `5857287`.

These plans are self-contained handoffs for an executor with zero context from
the survey session. Each executor: read the plan fully before starting, run the
drift check, honor STOP conditions, and update your row below when done.

The repo is in good shape overall — zero-dependency hosted service with a real
test suite, strict ID validation, defended path traversal, a sandboxed render
route, and timing-safe token comparison. These plans are hardening and public-repo
hygiene, not emergency fixes.

## Selection rationale

Written non-interactively. These five are the public-release-relevant findings by
leverage. The set covers: the two real security surfaces exposed by going public
(deep-link host/token redirect; host-header link injection), the one confirmed
correctness bug (verifier digest divergence on whitespace-padded annotations), and
first-impression contributor hygiene. Direction ideas (MCP server, `npx publish`,
GitHub Action) were surfaced to the maintainer separately and are **not** planned
here — they are net-new product scope, not release readiness.

## Findings considered and rejected

- **"Missing `await` on S3 `deletePayload`"** — rejected: the `await` is present
  (`storage.js:152` inside `readPayloadByKey`, awaited again at `storage.js:164`);
  delete correctly returns `false` for a missing record. Subagent misread.
- **"No SECURITY.md / CODE_OF_CONDUCT.md"** — rejected: both already exist at the
  repo root.
- **Path traversal in storage / static serving** — rejected: IDs are validated by
  `safeID` (`/^[A-Za-z0-9_-]{8,80}$/`); static serving uses `path.normalize` +
  `startsWith(rootDirectory)` + `.playpen-store` block. Properly defended.
- **S3 read-error response-body leak (`storage.js`)** — rejected: Node GC reclaims
  the small unconsumed body; not worth the code churn.
- **No ESLint/Prettier/typecheck gate (DX)** — rejected for now: the repo is
  zero-dependency by deliberate design; adding a linter's dependency cost outweighs
  the benefit for a codebase this size. Documenting style in CONTRIBUTING (folded
  into plan 005's spirit) is the lighter option if desired later.
- **Token-comparison timing leak via early length check (`server.js:762`)** —
  considered, not planned: real but very low severity (leaks only token *length*,
  only when a token is configured, behind network jitter). Fold into 003 if
  touching that file, otherwise not worth a standalone plan.
- **Markdown render unbounded by line count (hosted viewer)** — considered, not
  planned: already bounded by the 2 MB publish cap and runs client-side in the
  viewer only. Low leverage.
- **S3 driver only tested against a hand-written fake** — considered, not planned
  as release-blocking: a real MinIO/S3 integration test needs Docker in CI, which
  conflicts with the zero-dep/simple-CI posture. Worth revisiting before the first
  production S3 deployment; noted here so it isn't re-audited as "missing."
- **No native Swift test target** — acknowledged, not planned: deliberately
  deferred per PLAN.md phasing; would be an L-effort new target. Plans 001/002/004
  each note where a test would slot in *if* a target is added later.

## Direction (not planned — maintainer's call)

Grounded forward options surfaced during the survey, each scoped as a design/spike
if pursued:

- **MCP server** wrapping the existing OpenAPI (`PlayPen/Hosted/openapi.json`) —
  the project's whole pitch is "agents publish artifacts here" (`AGENTS.md`), yet
  there is no MCP entry point. Strongest-grounded option.
- **`npx playpen publish`** — `publish-file.js` is already standalone; packaging it
  removes the "clone the repo to publish" barrier for agents in sandboxes.
- **GitHub Action** to publish an artifact and comment the link on a PR — fits the
  stated QA/review audience; `.github/workflows/ci.yml` already exercises the
  service but nothing consumes it.
